# clangd Setup Notes

## Context

This project uses Clangd with `compile_commands.json` generated by CMake
(GCC preset). On Ubuntu-like systems, Clangd uses its own Clang-based parser
but relies on GCC's `libstdc++` headers for C++ standard library symbols.

When a newer GCC runtime package is installed without the corresponding
`libstdc++` development headers, Clangd auto-selects the newest GCC toolchain
and fails to resolve includes — even though the project builds fine with `g++`
(which is pinned to a working GCC version).

Typical error messages in Clangd output:

- `'iostream' file not found`
- `'cstdint' file not found`
- `use of undeclared identifier 'std'`

## Root Cause

Clangd scans `/usr/lib/gcc/<machine>/` at startup and picks the **highest**
GCC version it finds. If that version lacks the `libstdc++-<ver>-dev` package,
the C++ standard library headers are simply absent from Clangd's include path.

On this project's target environment the symptom was:

| Component        | Installed                         | C++ headers      |
|------------------|-----------------------------------|------------------|
| `g++` (GCC 15)   | yes                               | `/usr/include/c++/15/` |
| `gcc-16` runtime | yes (`gcc-16`, `gcc-16-base`, …) | **missing**       |

Clangd chose GCC 16 → no headers → `fatal error: 'iostream' file not found`.

## Fix

Install the matching `libstdc++` development package:

```bash
sudo apt install libstdc++-16-dev
```

This provides `/usr/include/c++/16/` and Clangd resolves includes
automatically, with **no additional project-level configuration**.

## Project-Level Configuration (`.clangd`)

The `.clangd` file at the repository root has only the minimum required for
CEF-compatible parsing:

```yaml
CompileFlags:
  Add:
    - -Wno-gnu-zero-variadic-macro-arguments
  Remove:
    - -Werror
```

- `-Wno-gnu-zero-variadic-macro-arguments` prevents Clang from aborting on
  CEF headers that use GNU `##__VA_ARGS__` extensions.
- `-Werror` is stripped because it is intended for production GCC builds;
  Clangd should not treat pedantic CEF warnings as fatal parse errors.

No `Compiler`, `--driver-mode=g++`, or `--gcc-install-dir` overrides are
needed — Clangd discovers the correct toolchain automatically once the
system headers are in place.

## Verify Setup

1. Check that the matching headers are present:

```bash
ls /usr/include/c++/$(gcc -dumpversion 2>/dev/null || g++ -dumpversion)
```

2. Run a minimal Clangd check without extra flags:

```bash
CLANGD="$HOME/.config/Code/User/globalStorage/llvm-vs-code-extensions.vscode-clangd/install/*/clangd_*/bin/clangd"
echo '#include <iostream>' | "$CLANGD" --check=<(echo '#include <iostream>')
```

Expected: `All checks completed, 0 errors`.

3. In VS Code / Cursor:

- `Clangd: Restart language server`
- If symptoms persist: `Developer: Reload Window`

## Troubleshooting

- If headers are still missing, confirm the `libstdc++-<ver>-dev` package
  matches the highest GCC version visible to Clangd:

```bash
g++ -dumpmachine
ls -1 /usr/lib/gcc/$(g++ -dumpmachine)/
ls -1 /usr/include/c++/
```

Every GCC version listed in `/usr/lib/gcc/<machine>/` should have a
corresponding entry in `/usr/include/c++/`.

- If Clangd picks a version without headers and installing the dev package
  is not an option, add an explicit hint in `.clangd`:

```yaml
CompileFlags:
  Add:
    - --gcc-install-dir=/usr/lib/gcc/x86_64-linux-gnu/15
```

This is a last-resort workaround — prefer installing the matching
`libstdc++-<ver>-dev` package.
